1. Architecture technique
1.1 Extension du modèle multi-tenant
HERALDO AGENCE ajoute une couche d'abstraction supplémentaire au-dessus du modèle multi-tenant existant. Chaque compte AGENCE est un tenant HERALDO standard, mais il contient lui-même plusieurs sous-espaces (clients) qui sont des tenants logiques imbriqués.

Principe : Tenant AGENCE → N sous-tenants CLIENT. Chaque client d'une agence est un sous-tenant avec son propre schéma de données, ses propres clés d'intégration (OAuth réseaux sociaux), et ses propres paramètres de marque blanche. L'isolation est garantie au niveau base de données par le préfixe agency_{agency_id}_client_{client_id}_ sur tous les schémas clients.

	•	refresh_token TEXT ENCRYPTED NULLABLE
	•	token_expires_at TIMESTAMP NULLABLE

3. Endpoints API — Module M10
3.1 Gestion des clients agence
	•	GET /api/v1/agency/clients — Liste des clients avec stats résumées
	•	POST /api/v1/agency/clients — Créer un nouveau client
	•	GET /api/v1/agency/clients/{clientId} — Détail client + analytics
	•	PATCH /api/v1/agency/clients/{clientId} — Mettre à jour le profil client
	•	DELETE /api/v1/agency/clients/{clientId} — Archiver un client
	•	POST /api/v1/agency/clients/{clientId}/switch — Basculer vers ce client (met à jour le contexte de session)

3.2 Marque blanche
	•	GET /api/v1/agency/clients/{clientId}/whitelabel — Configuration marque blanche
	•	PUT /api/v1/agency/clients/{clientId}/whitelabel — Mettre à jour (nom, email, logo)
	•	POST /api/v1/agency/clients/{clientId}/whitelabel/verify-dns — Vérifier la configuration DNS pour l'email personnalisé

3.3 Réseaux sociaux
	•	GET /api/v1/agency/clients/{clientId}/social-accounts — Comptes connectés
	•	POST /api/v1/agency/clients/{clientId}/social-accounts/connect — Initier OAuth pour un réseau
	•	DELETE /api/v1/agency/clients/{clientId}/social-accounts/{accountId} — Déconnecter
	•	POST /api/v1/agency/clients/{clientId}/social-posts — Créer et planifier une publication
	•	GET /api/v1/agency/clients/{clientId}/social-posts — Historique des publications
	•	POST /api/v1/agency/clients/{clientId}/social-posts/{postId}/publish — Publication immédiate
	•	GET /api/v1/agency/clients/{clientId}/social-posts/{postId}/stats — Engagement stats

3.4 Rapports clients
	•	GET /api/v1/agency/clients/{clientId}/reports — Historique des rapports
	•	POST /api/v1/agency/clients/{clientId}/reports/generate — Générer un rapport pour une période
	•	GET /api/v1/agency/clients/{clientId}/reports/{reportId}/pdf — Télécharger le PDF
	•	POST /api/v1/agency/clients/{clientId}/reports/{reportId}/send — Envoyer au client

3.5 Suivi du temps et facturation
	•	GET /api/v1/agency/clients/{clientId}/time-entries — Entrées de temps
	•	POST /api/v1/agency/clients/{clientId}/time-entries — Ajouter une entrée
	•	POST /api/v1/agency/clients/{clientId}/time-entries/start — Démarrer un chronomètre
	•	POST /api/v1/agency/clients/{clientId}/time-entries/{id}/stop — Arrêter le chronomètre
	•	POST /api/v1/agency/clients/{clientId}/invoices/generate — Générer une facture mensuelle

3.6 Webhook validation client
	•	GET /api/v1/agency/clients/{clientId}/review/{token} — Interface de validation (accès sans compte)
	•	POST /api/v1/agency/clients/{clientId}/review/{token}/approve — Approuver un contenu
	•	POST /api/v1/agency/clients/{clientId}/review/{token}/reject — Rejeter avec commentaire


4. Intégrations OAuth réseaux sociaux
4.1 Architecture OAuth sécurisée
Les tokens OAuth sont la donnée la plus sensible du module AGENCE. Une compromission donnerait accès aux comptes officiels des clients sur tous les réseaux sociaux. L'architecture de stockage est donc renforcée au-delà du standard.

Chiffrement des tokens
	•	Chaque token est chiffré avec AES-256-GCM avant stockage en base.
	•	La clé de chiffrement est dérivée de la clé maître de l'agency_id + client_id — spécifique à chaque client.
	•	La clé maître n'est jamais stockée en base — elle est reconstituée à runtime depuis un secret vault.
	•	Même en cas de dump de la base de données, les tokens sont illisibles sans accès au vault.

Gestion de l'expiration
	•	Vérification du token_expires_at avant chaque appel API réseau social.
	•	Rafraîchissement automatique du token si expiration < 1 heure (Bull job background).
	•	Si le rafraîchissement échoue : notification immédiate à l'AGENCY_ADMIN et désactivation du compte social en attente de reconnexion.
	•	Rotation de token logguée dans l'audit log avec timestamp.

5.2 Gestion offline
Un attaché de presse sur le terrain n'a pas toujours du réseau. L'application doit rester utilisable en mode dégradé.

Données synchronisées en cache local
	•	Annuaire journalistes de tous les clients — mis à jour à chaque lancement en ligne.
	•	Agenda des 14 prochains jours — synchronisé toutes les heures.
	•	Derniers communiqués (30 jours) — consultables hors ligne.
	•	Profils clients avec logos et couleurs — stockés localement.

Actions disponibles hors ligne
	•	Composition d'un communiqué (mis en file d'attente pour envoi au retour du réseau).
	•	Consultation et modification des contacts journalistes.
	•	Prise de notes vocales (transcrites au retour du réseau).
	•	Consultation de l'agenda et ajout d'événements (synchronisés au retour).

Conflits de synchronisation
	•	Stratégie last-write-wins pour les modifications de contacts.
	•	Notification à l'utilisateur si conflit détecté lors de la synchronisation.

5.3 Sécurité mobile
	•	Authentification biométrique (Face ID / Touch ID) — fallback PIN 6 chiffres.
	•	Token JWT stocké dans MMKV chiffré — jamais dans AsyncStorage non chiffré.
	•	Session expirée après 30 minutes d'inactivité — réauthentification requise.
	•	Certificate pinning sur toutes les requêtes API — protection contre MITM.
	•	Détection de jailbreak/root — désactivation de l'app sur appareils compromis.
	•	Screenshot protection sur les écrans sensibles (INBOX, CRM, détail client).


6. Génération des rapports PDF
6.1 Architecture de génération
La génération des rapports PDF est un processus asynchrone déclenché en arrière-plan par un Bull job. Le PDF est généré côté serveur (Node.js), stocké dans MinIO, et l'URL signée est renvoyée au client.

Stack de génération
	•	Puppeteer headless : rendu HTML → PDF haute qualité avec gestion des images, couleurs et typographies des clients.
	•	Template HTML Handlebars : template paramétré avec les données du rapport (KPIs, graphiques Chart.js, logo client).
	•	Chart.js rendu côté serveur : graphiques vectoriels intégrés au PDF (pas d'images bitmap).
	•	Watermark conditionnel : si formule SOLO, mention discrète "Powered by HERALDO" — supprimée en PRO et AGENCE.

Données incluses dans le rapport
	•	Page de garde : logo client, période, nom de l'agence ou de l'attaché.
	•	Synthèse exécutive : 4 KPIs clés avec icônes et comparaison mois précédent.
	•	Communiqués : liste avec objet, date, nombre de destinataires, taux de lecture, portée estimée.
	•	Top journalistes : classement des 5 journalistes les plus actifs avec photo et média.
	•	Réseaux sociaux : publications du mois, engagement total par réseau, graphique d'évolution.
	•	Mentions VEILLE : synthèse des mentions si module activé.
	•	Recommandations : 3 actions générées automatiquement par analyse des KPIs.
	•	Annexe : journal complet de toutes les actions du mois.
