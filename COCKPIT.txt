2.6 Règles d'adaptation automatique par plateforme
Facebook
Longueur max 63 206 caractères, pas de troncature automatique.
Images : 1 à 10 images par post, format JPEG/PNG/GIF, max 30 Mo par image.
Vidéos : max 10 Go, formats MP4/MOV, durée max 4h.
Hashtags : illimités mais ne pas mettre plus de 5 dans le contenu visible, surcharger est pénalisé algorithmiquement.
Instagram
Longueur max 2 200 caractères. Troncature à partir de 125 caractères dans le feed (UI).
Images : format carré 1:1 préféré (1080×1080), portrait 4:5 accepté. Converter automatiquement si le fichier source est 16:9.
Vidéos Reels : max 90 secondes, format 9:16 (1080×1920), conversion automatique si nécessaire.
Hashtags : jusqu'à 30 autorisés, recommandé entre 5 et 11. Placer en premier commentaire pour ne pas encombrer le texte principal.
LinkedIn
Longueur max 3 000 caractères. Troncature à 210 caractères dans le feed (UI).
Images : 1200×627 pixels recommandé pour les articles, 1080×1080 pour posts simples.
Ton : formel et professionnel, éviter les emojis abondants.
Hashtags : 3 à 5 maximum, placés en fin de texte.
2.7 Workflow de validation
Deux modes de publication possibles selon le paramètre approval_required :
Mode direct (approval_required = false)
L'attaché ou l'utilisateur ayant droit publish crée et publie directement.
Statut passe immédiatement de draft à scheduled (si date future) ou publishing (si immédiat).
Mode validation (approval_required = true)
L'attaché crée, statut devient pending_approval.
Le DG / Maire (utilisateur ayant droit approve) reçoit notification email + push app.
Le validateur accepte (statut → scheduled) ou refuse (statut → draft avec commentaire).
La publication automatique ne se déclenche QUE si status = scheduled au moment du scheduled_at.
2.8 Worker de publication planifiée
Un worker Celery ou NestJS Scheduler tourne toutes les 60 secondes et exécute :

1. SELECT social_posts WHERE status='scheduled' AND scheduled_at <= NOW()
2. Pour chaque post :
   a. Marquer status = 'publishing'
   b. Pour chaque target_account_id :
      - Récupérer access_token (déchiffrer)
      - Adapter le contenu au format de la plateforme
      - Appeler l'API externe (Facebook Graph, Instagram, LinkedIn)
      - Enregistrer external_post_id et external_post_url
      - Si succès : status = 'published'
      - Si échec : status = 'failed', error_log renseigné, retry 3x à 5 min d'intervalle
3. Notifier l'utilisateur créateur (email + push)
2.9 Règles de gestion
RG-SOCIAL-01 : Un compte social ne peut être connecté que par un utilisateur ayant droit manage_social_accounts dans sa matrice de permissions.
RG-SOCIAL-02 : Les access tokens sont chiffrés AES-256 avec clé distincte de celle utilisée pour les autres données sensibles (rotation indépendante).
RG-SOCIAL-03 : Une publication échouée génère une alerte Slack/email à l'équipe HERALDO support si 3 retries ont échoué.
RG-SOCIAL-04 : Les tokens Facebook doivent être rafraîchis avant expiration via un worker quotidien (Facebook émet des tokens 60 jours, rafraîchir à J-7).
RG-SOCIAL-05 : Le nombre de publications simultanées par compte est limité à 100/jour pour éviter le bannissement par la plateforme.
RG-SOCIAL-06 : Pas de publication de contenu identique sur le même compte dans les 24h (anti-spam).
RG-SOCIAL-07 : Les publications supprimées côté plateforme externe marquent le post HERALDO comme orphan mais ne suppriment pas l'entrée en BDD.

4.4 Format de description des templates (template_json)
Chaque template est défini par un JSON qui décrit les zones éditables. Exemple complet :

{
  "width": 1080,
  "height": 1080,
  "background": {
    "type": "color",  // "color" | "image" | "gradient"
    "value": "{{brand.color_primary}}"  // ou couleur hex fixe
  },
  "elements": [
    {
      "type": "image",
      "id": "user_image",
      "source": "{{params.main_image}}",  // uploadée par l'utilisateur
      "x": 0, "y": 0, "width": 1080, "height": 720,
      "fit": "cover",
      "overlay": { "color": "#000000", "opacity": 0.3 }
    },
    {
      "type": "text",
      "id": "title",
      "content": "{{params.title}}",
      "x": 60, "y": 780, "width": 960, "height": 140,
      "font": "{{brand.font_heading}}",
      "font_size": 56, "font_weight": "bold",
      "color": "#FFFFFF",
      "align": "left", "valign": "middle",
      "max_length": 80  // troncature automatique avec ...
    },
    {
      "type": "image",
      "id": "logo",
      "source": "{{brand.logo_white_url}}",
      "x": 60, "y": 60, "width": 200, "height": 80,
      "fit": "contain"
    },
    {
      "type": "shape",
      "id": "accent_bar",
      "shape": "rectangle",
      "x": 60, "y": 940, "width": 100, "height": 6,
      "fill": "{{brand.color_accent}}"
    }
  ],
  "editable_fields": [
    { "id": "title", "label": "Titre principal", "type": "text", "required": true, "max": 80 },
    { "id": "main_image", "label": "Image de fond", "type": "image", "required": true }
  ]
}
4.5 Moteur de rendu — Algorithme
Fonction principale : renderVisual(templateId, brandKit, params) → visualUrl
1. Charger le template_json depuis studio_templates
2. Charger le brand_kit du client
3. Substituer toutes les variables {{brand.xxx}} et {{params.xxx}} dans le JSON
4. Créer un canvas node-canvas aux dimensions du template
5. Pour chaque élément dans elements[] :
   a. Si type=background : fillRect ou drawImage selon sous-type
   b. Si type=image : loadImage + drawImage avec fit (cover/contain) + overlay si présent
   c. Si type=text : setFont, measureText, wrap text si nécessaire, fillText avec align/valign
   d. Si type=shape : fillRect ou beginPath selon shape
6. Exporter en PNG ou JPEG (option quality)
7. Upload S3 avec key studio/{client_id}/{visual_id}.{ext}
8. Retourner l'URL CDN publique

Nouvelle vue matérialisée crm_journalist_scores — Score relationnel
Vue rafraîchie chaque nuit (00:00) par un worker pour calculer un score de relation par journaliste et par client.
Champ calculé
Formule
interactions_last_90d
COUNT(crm_interactions) WHERE occurred_at > NOW()-90 DAYS
last_interaction_at
MAX(occurred_at)
publications_generated
COUNT WHERE interaction_type='publication_detected'
fcm_total_paid_fcfa
SUM WHERE interaction_type='fcm_paid'
relation_score
0-100, formule pondérée (voir section 6.3)
relation_tier
cold | warm | hot | engaged (selon score)

6.3 Formule de calcul du score de relation
Score 0-100 calculé par la formule :

  relation_score = 
    min(100,
      (interactions_last_90d * 3)            // jusqu'à 30 pts
      + (publications_generated * 8)          // jusqu'à 40 pts
      + (fcm_total_paid_fcfa / 10000)         // jusqu'à 20 pts
      + (last_interaction_bonus)              // 0 à 10 pts
    )

  last_interaction_bonus =
    si last_interaction_at > NOW()-7d  → 10
    si last_interaction_at > NOW()-30d → 6
    si last_interaction_at > NOW()-90d → 3
    sinon                              → 0
Tier déduit du score :
0–24 : cold (relation inexistante ou dormante)
25–49 : warm (contact régulier mais pas de résultats)
50–74 : hot (relation active, publications régulières)
75–100 : engaged (journaliste fidèle, FCM actif, publications fréquentes)

11.3 Pattern de code à respecter
Pour cohérence avec le code existant du projet HERALDO PRESS :
Un module NestJS par module fonctionnel : src/modules/social/, src/modules/agenda/, src/modules/studio/, etc.
Structure de module standard : controller.ts, service.ts, dto/, entities/, tests/
Validation via class-validator sur les DTOs d'entrée
Transformation via class-transformer pour les réponses
Guards JWT sur toutes les routes sauf webhooks publics
Decorators custom @CurrentUser(), @RequirePermission(...) déjà existants à réutiliser
Gestion d'erreur via exceptions NestJS typées (BadRequestException, NotFoundException, etc.)
Tous les async externes (API calls, DB) wrappés par try/catch avec logging structuré (Pino)
