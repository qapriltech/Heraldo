1. Architecture générale
1.1 Principe d'architecture
HERALDO est une application web SaaS multi-tenant isolée. Chaque institution cliente dispose d'une instance logique séparée avec sa propre base de données, son propre espace de stockage et son propre sous-domaine. L'isolation est logique en V1 et physique (on-premise) pour HERALDO SOUVERAIN.

1.2 Architecture applicative — Vue d'ensemble
HERALDO suit une architecture en 3 couches : Frontend (client web), Backend (API REST + services), Infrastructure (base de données, stockage, queues).

1.3 Architecture multi-tenant
HERALDO utilise un modèle multi-tenant avec isolation par schéma PostgreSQL (schema-per-tenant). Chaque institution dispose de son propre schéma dans la base de données, avec ses propres tables, index et séquences. L'application routing détermine le tenant via le sous-domaine ou un header HTTP spécifique.

Structure des sous-domaines
	•	Format : {institution}.heraldo.ci — ex: primature.heraldo.ci, mint.heraldo.ci
	•	Certificat SSL wildcard *.heraldo.ci géré via Let's Encrypt + Certbot.
	•	Fallback : heraldo.ci/{institution} pour les accès sans sous-domaine configuré.

Isolation des données
	•	Chaque tenant a son propre schéma PostgreSQL — aucune requête ne peut traverser les schémas sans privilège explicite.
	•	Les fichiers MinIO sont organisés en buckets séparés par tenant_id avec ACL strictes.
	•	Les clés de chiffrement Redis sont préfixées par tenant_id — impossible d'accéder aux données d'un autre tenant.

4. Sécurité
4.1 Authentification et autorisation
JWT + Refresh Token
	•	Access token : durée de vie 15 minutes — JWT signé RS256 avec clé privée par tenant.
	•	Refresh token : durée de vie 7 jours — stocké en base, révocable, rotation à chaque usage.
	•	Double authentification (2FA) : TOTP via Google Authenticator / Authy — obligatoire pour les rôles Admin et SG.
	•	SSO optionnel : intégration SAML 2.0 pour les institutions ayant un Active Directory.

4.2 Sécurité des données
Chiffrement
	•	Données en transit : TLS 1.3 obligatoire — TLS 1.2 accepté pour compatibilité, TLS 1.0/1.1 bloqués.
	•	Données au repos : chiffrement AES-256-GCM sur les fichiers sensibles (documents confidentiels, pièces jointes NV2/NV3).
	•	Mots de passe : bcrypt avec salt factor 12 — jamais stockés en clair.
	•	Secrets applicatifs : stockés dans des variables d'environnement via HashiCorp Vault ou équivalent.

Protection des APIs
	•	Rate limiting : 100 requêtes/minute par IP, 1000/minute par token authentifié.
	•	CORS : whitelist stricte des origines autorisées par tenant.
	•	CSRF protection : tokens CSRF pour toutes les mutations d'état.
	•	Input validation : validation stricte de tous les inputs via class-validator (NestJS).
	•	SQL injection : prévention via ORM + requêtes paramétrées exclusivement.
	•	XSS : sanitization DOMPurify côté client + Content Security Policy headers.

Audit et logging
	•	Chaque action utilisateur (connexion, envoi, modification, export) est loggée dans un journal d'audit immuable.
	•	Format de log structuré JSON : timestamp, user_id, tenant_id, action, ip, user_agent, payload_hash.
	•	Rétention des logs : 12 mois minimum, archivage froid après 3 mois.
	•	Alertes de sécurité automatiques : tentatives de connexion multiples, accès hors plage horaire, exports massifs.

5.2 CI/CD — Pipeline de déploiement
	•	Développeur push sur branche feature → Tests automatiques (Jest + Cypress).
	•	PR validée par 2 reviewers → Merge sur develop → Déploiement auto sur staging.
	•	QA validée sur staging → Merge sur main → Déploiement prod avec zero-downtime (blue-green).
	•	Rollback automatique si les health checks échouent dans les 5 minutes post-déploiement.

Tests obligatoires
	•	Tests unitaires : couverture minimum 80% sur les services métier critiques.
	•	Tests d'intégration : tous les endpoints API testés avec données réelles.
	•	Tests E2E : parcours utilisateur critiques (envoi communiqué, login, analytics).
	•	Tests de charge : simulation de 1000 utilisateurs simultanés avant chaque release majeure.

6. Spécifications techniques modules spéciaux
6.1 Module LIVE — Architecture WebRTC
HERALDO LIVE utilise MediaSoup comme Selective Forwarding Unit (SFU) pour la diffusion vidéo. Ce choix permet de gérer jusqu'à 500 spectateurs simultanés sans dégradation de qualité, tout en conservant la latence ultra-faible du WebRTC (< 500ms).

Flux de connexion
	•	L'animateur démarre la session LIVE via le dashboard HERALDO — MediaSoup crée une room sécurisée.
	•	HERALDO génère un token JWT unique pour chaque journaliste accrédité et l'envoie par notification.
	•	Le journaliste ouvre son navigateur, HERALDO charge le client WebRTC et négocie la connexion avec MediaSoup.
	•	MediaSoup reçoit le flux de l'animateur et le redistribue en mode SFU vers tous les spectateurs connectés.
	•	Socket.io gère le canal de données parallèle (questions, chat, présence).

Qualité et adaptive bitrate
	•	Résolution de diffusion : 1080p par défaut, adaptation automatique selon la bande passante (720p, 480p, 360p).
	•	Codec vidéo : VP8 (compatible tous navigateurs) avec fallback H.264.
	•	Codec audio : Opus 48kHz — qualité broadcast.
	•	Délai de latence cible : < 500ms pour le direct, 2-5 secondes pour le HLS fallback sur connexions lentes.

6.2 Module VEILLE — Architecture de scraping
HERALDO VEILLE utilise une architecture de micro-services pour surveiller les sources médiatiques en temps réel. Chaque source a son propre collector indépendant, permettant d'ajouter ou modifier des sources sans redéploiement global.

Architecture des collectors
	•	Collector RSS/Atom : scraping des flux RSS de 50+ médias ivoiriens toutes les 5 minutes.
	•	Collector Social Media : API Twitter v2 + Facebook Graph API — polling toutes les 2 minutes.
	•	Collector Web Scraper : Playwright headless pour les sites sans API ni RSS — toutes les 15 minutes.
	•	Collector WhatsApp : intégration WhatsApp Business API pour les canaux publics officiels.

Pipeline de traitement
	•	Réception de la mention brute dans la queue Bull.
	•	Déduplication : hash du contenu pour éviter les doublons cross-sources.
	•	Analyse sémantique via API OpenAI (GPT-4o-mini) : détection des entités surveillées, tonalité, urgence.
	•	Score de pertinence calculé (0-100) selon les mots-clés configurés par le tenant.
	•	Si score > seuil → notification push temps réel via Socket.io + email si activé.
	•	Stockage dans PostgreSQL + indexation Elasticsearch pour la recherche.
