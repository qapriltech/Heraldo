1. PHILOSOPHIE DE L'ARCHITECTURE

HERALDO PRESS fait tourner deux systèmes qui communiquent. Odoo gère tout ce qui est gestion d'entreprise standard (comptabilité, facturation, paie, CRM prospects). La plateforme HERALDO gère tout ce qui est spécifique au métier SaaS de communication institutionnelle (abonnements clients, usage, journalistes accrédités, FCM, reporting métier). Les deux systèmes communiquent par API pour automatiser les flux de bout en bout.

Pour éviter toute confusion entre les équipes et dupliquer des efforts, nous posons une règle simple :

1.2 Principe d'intégration
Les deux systèmes ne se substituent pas l'un à l'autre. Ils se complètent via des flux d'information automatisés qui évitent la double saisie et les erreurs. Le principe directeur : chaque donnée a une seule source de vérité, et les autres systèmes la récupèrent par API.
Exemples concrets de flux automatisés :
Une institution cliente souscrit à HERALDO MAIRIE → HERALDO déclenche la création d'un client dans Odoo + émission de la facture + envoi email avec lien de paiement.
Un paiement FCM est effectué à un journaliste → HERALDO enregistre la transaction + génère une ligne de commission (15%) → exporte quotidiennement vers Odoo en revenu variable.
Un salarié est recruté → Odoo gère le contrat, les bulletins de paie, les congés. HERALDO gère uniquement son accès à la plateforme selon son rôle.
Un impayé est détecté dans Odoo → API notifie HERALDO qui restreint automatiquement les accès après 30 jours.

3.5 Workers automatisés
Worker 1 — Préavis de renouvellement (tous les jours à 8h)
Sélectionner les abonnements active dont end_date est entre J+55 et J+65.
Marquer status = renewing.
Générer la facture de renouvellement dans Odoo via API.
Envoyer email au client avec lien vers espace de paiement + PDF facture.
Notifier le Customer Success assigné pour suivi.
Worker 2 — Détection d'impayés (tous les jours à 10h)
Interroger Odoo via API pour lister les factures overdue (> 30 jours).
Croiser avec les abonnements HERALDO actifs.
Pour chaque match : passer subscription à status = suspended, envoyer email au client.
Restreindre les accès plateforme automatiquement (lecture seule).
Worker 3 — Expiration automatique (tous les jours à 2h)
Sélectionner les abonnements renewing dont end_date est dépassée de 7 jours sans paiement.
Passer à status = expired.
Désactiver tous les accès plateforme.
Notifier Customer Success pour appel de fin de relation.

6. MODULE A4 — FCM-ADMIN (ADMINISTRATION FCM)

6.1 Objectif fonctionnel
Superviser l'intégralité des flux financiers FCM depuis l'administration HERALDO : alimentations de fonds par les clients, configurations d'événements, validations de preuves, déclenchements de paiements Mobile Money, suivi des commissions HERALDO. Ce module est critique car il manipule des transactions financières réelles et doit être totalement auditable.
6.2 Architecture du flux FCM
Rappel des flux financiers FCM (vue administration) :
Flux 1 — Client alimente son portefeuille FCM (paiement entrant via Mobile Money ou virement)
Flux 2 — Client configure un événement FCM (montant par preuve, nombre max de preuves, conditions)
Flux 3 — Journaliste soumet une preuve → attend validation
Flux 4 — Preuve validée par le client ou HERALDO → déclenchement paiement Mobile Money au journaliste
Flux 5 — HERALDO prélève sa commission 15% automatiquement → comptabilisée dans Odoo
6.3 Tables critiques à administrer
La plateforme HERALDO maintient déjà des tables fcm_* définies dans CDC-HERALDO-001. Le module A4 ajoute des **vues et endpoints d'administration** pour l'équipe HERALDO :

6.5 Règles de gestion FCM-ADMIN
RG-FCM-ADMIN-01 : Toute action admin sur FCM (arbitrage, retry paiement, résolution contestation) est loggée dans audit_logs avec l'admin responsable, la date, et le motif.
RG-FCM-ADMIN-02 : Un admin ne peut pas modifier le montant d'une preuve validée ni le montant d'une commission calculée — ces valeurs sont immuables une fois l'événement conclu.
RG-FCM-ADMIN-03 : En cas de contestation d'un journaliste sur une preuve rejetée, HERALDO dispose de 72h pour trancher. Au-delà, escalation automatique au Directeur Général.
RG-FCM-ADMIN-04 : Les paiements Mobile Money échoués sont automatiquement retentés 3 fois (à 1h, 6h, 24h) avant alerte humaine.
RG-FCM-ADMIN-05 : Les commissions HERALDO sont exportées chaque nuit vers Odoo comme revenus variables, avec ligne analytique par client pour reporting par segment.

8. MODULE A6 — SUPPORT (HELPDESK INTÉGRÉ)

8.1 Objectif fonctionnel
Gérer les demandes d'assistance des clients (institutions) et des journalistes via un système de tickets intégré à la plateforme. Au démarrage, nous utilisons l'outil externe Crisp. Ce module peut être construit en Phase 2 quand le volume justifiera une solution intégrée pour économiser l'abonnement externe et centraliser les données support avec les données client.
8.2 Stratégie en deux temps
Phase 1 — Outil externe (Crisp ou équivalent) — M0 à M12
Widget Crisp intégré dans la plateforme HERALDO via JS.
Coût : 5 000 à 15 000 FCFA/mois selon nombre d'agents.
Gestion simple, pas de développement nécessaire.
Phase 2 — Support intégré — M12+
Module construit dans HERALDO avec tables tickets, ticket_messages, ticket_attachments.

10.3 Architecture technique de l'intégration
Méthode recommandée
Odoo expose une API XML-RPC et JSON-RPC standard. Toutes les opérations CRUD sont accessibles. Documentation officielle : https://www.odoo.com/documentation
Côté HERALDO
Module NestJS src/modules/odoo-sync/ avec service OdooClientService qui encapsule les appels API.
Config via variables d'environnement : ODOO_URL, ODOO_DB_NAME, ODOO_API_USER, ODOO_API_PASSWORD.
Queue Bull/Redis pour les jobs sortants (résilience en cas de panne Odoo temporaire).
Worker dédié consumant la queue avec retry automatique (3 tentatives à 5, 30, 300 secondes).
Endpoint webhook POST /v1/webhooks/odoo pour recevoir les événements entrants.
Côté Odoo
Module Python custom à développer (par intégrateur Odoo CI) qui émet les webhooks vers HERALDO lors des événements financiers clés.
Budget intégration : environ 3 à 5 millions FCFA pour un intégrateur Odoo CI qualifié (développement + formation + recette).

10.6 Robustesse
Idempotence : chaque opération sortante porte un external_reference unique, si Odoo reçoit deux fois le même ID, il le détecte et refuse la duplication.
Retry automatique : 3 tentatives à 5s, 30s, 300s puis alerte humaine.
Buffer local : les événements qui échouent à synchroniser sont stockés dans odoo_sync_queue et retentés par un worker toutes les 10 minutes jusqu'à succès.
Réconciliation nocturne : chaque nuit, un script vérifie que toutes les souscriptions HERALDO ont leur pendant Odoo et signale les écarts.
